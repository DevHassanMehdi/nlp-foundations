Overview

What is this project?

You type a prompt — like "a sci-fi thriller on a space station" — and the system writes a full TV episode draft. Title, characters, logline, plot outline, scenes, and a sample script.

Three different AI models generate the same episode at once, so you can see how each one performs side by side.

The Goal

Build a complete, local NLP pipeline — from raw data to a live web demo — comparing old and new generation methods.

The Constraint

No paid APIs. Everything runs locally on my own machine. Fully reproducible from training to inference.

System Design

How it's built

The project is split into five clean layers, each with a single job.

01

Data Layer

Cleans and tokenizes the text corpus used for training.

02

Model Layer

N-gram baseline, Word-LSTM, and a local transformer model (Qwen2.5 Instruct).

03

Structured Generation Layer

A planner creates section prompts; a formatter assembles them into a full episode.

04

API Layer

FastAPI backend exposes a /generate endpoint for real-time inference.

05

Frontend Layer

Live web UI shows all three model outputs side by side with prompt and temperature controls.

Development Journey

How the project evolved

v1

N-gram + Char-RNN Baseline

Fast to build. Outputs were text-shaped but mostly incoherent — many fake words and broken sentences.

v2

Switch to Word-LSTM

Real words only. Added temperature, top-k sampling, and repetition penalties. Major leap in readability.

v3

Planner + Formatter

Instead of generating one big block of text, the system now fills each episode section one at a time. Outputs became consistently structured.

v4

Transformer Integration

Added Qwen2.5 Instruct as a third pipeline. Fixed prompt leakage and template compatibility issues. Best fluency of all models.

Problem Solving

What broke — and how I fixed it

🔤 Gibberish words

Char-level models invented fake words.
Fix: Moved to word-level LSTM. Problem mostly gone.

🔁 Repetition loops

Models kept repeating the same phrase.
Fix: Repetition penalty + no-repeat n-gram blocking.

📉 Story drift

Long outputs lost the plot entirely.
Fix: Section-by-section generation with a planner. Local coherence improved a lot.

🔧 Checkpoint crashes

Changing the model architecture broke saved weights.
Fix: Save the model config alongside the checkpoint.

💬 Prompt leakage

The transformer repeated its own instructions in the output.
Fix: Decode only new tokens and strip instruction echoes.

⚡ Training instability

Loss spiked unpredictably on long runs.
Fix: AdamW + learning-rate scheduling + gradient accumulation.

What I Learned

Six takeaways

01

Architecture > tuning

Switching model type had far more impact than tweaking hyperparameters.

02

Structure rescues weak models

The planner layer made weaker generators produce usable output.

03

More training ≠ better stories

After a point, longer runs gave diminishing gains in coherence.

04

Engineering reliability matters

Small issues in checkpoints or tokenizers can silently break everything.

05

Quality is hard to measure

A section can look good alone but fail as part of a full episode.

06

Integration details matter

How you format prompts and clean outputs shapes the final result more than expected.

What's Next

If I had more time

The biggest gains would come from better data and smarter pipelines, not just more training epochs.

→

Cleaner domain-specific dataset

A curated script corpus with strong narrative signals would improve all models immediately.

→

Fine-tune on episode/script data

A stronger instruct model fine-tuned specifically on screenwriting would change the output quality dramatically.

→

Cross-section consistency checks

A post-generation step that checks character names, events, and timelines across all sections.

→

Regenerate-on-failure loops

Automatic quality gates that detect weak sections and regenerate them before showing output to the user.

Conclusion

The project delivered a full working system: data pipeline, trained models, structured generation, FastAPI backend, and a live comparison UI.
